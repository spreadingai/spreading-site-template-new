# 介绍
Novel.sh 由两部分组成： 一个简约的静态站点生成器 （打开新窗口）具有 Vue 支持的主题系统和插件 API，以及为编写技术文档而优化的默认主题。它的创建是为了支持 Vue 自己的子项目的文档需求。

Novel.sh 生成的每个页面都有自己的预渲染静态 HTML，提供出色的加载性能并且对 SEO 友好。然而，一旦页面加载，Vue 就会接管静态内容并将其转换为完整的单页应用程序 (SPA)。当用户浏览网站时，会根据需要获取额外的页面。

## 怎么运行的
Novel.sh 网站实际上是一个由Vue提供支持的 SPA （打开新窗口）、Vue路由器 （打开新窗口）和网络包 （打开新窗口）。如果您以前使用过 Vue，那么您在编写或开发自定义主题时会注意到熟悉的开发体验（您甚至可以使用 Vue DevTools 来调试您的自定义主题！）。

在构建过程中，我们创建应用程序的服务器渲染版本，并通过虚拟访问每个路由来渲染相应的 HTML。这种方法的灵感来自Nuxt （打开新窗口）的nuxt generate命令和其他项目，如Gatsby （打开新窗口）。

每个 Markdown 文件都使用markdown-it编译为 HTML （打开新窗口）然后作为Vue组件的模板进行处理。这允许您直接在 Markdown 文件中使用 Vue，并且当您需要嵌入动态内容时非常有用。

## 为什么不 ...？
努克斯特
Nuxt 能够完成 Novel.sh 所做的事情，但它是为构建应用程序而设计的。Novel.sh 专注于以内容为中心的静态网站，并提供为开箱即用的技术文档量身定制的功能。

文档化/文档化
两者都是很棒的项目，而且都是由 Vue 驱动的。但它们都是完全运行时驱动的，因此不利于 SEO。如果您不关心 SEO 并且不想搞乱安装依赖项，那么这些仍然是不错的选择。

赫索
Hexo 一直很好地为 Vue 文档提供服务 - 事实上，我们的主站点距离迁移它可能还有很长的路要走。最大的问题是它的主题系统是静态的并且基于字符串——我们希望利用 Vue 来实现布局和交互性。此外，Hexo 的 Markdown 渲染配置起来也不是最灵活的。

吉特书
我们的大部分子项目文档都使用 GitBook。GitBook 的主要问题是它的开发重载性能对于大量文件来说是无法忍受的。默认主题也有一个相当有限的导航结构，而且主题系统也不是基于 Vue 的。GitBook 背后的团队也更专注于将其变成商业产品而不是开源工具。
